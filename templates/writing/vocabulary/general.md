# Vocabulary: general

Use plain, widely-understood software vocabulary. This profile suits documents
for readers outside the engine's own domain, where terms of art would obscure
rather than clarify.

Guidance:

- Prefer common words: "function", "setting", "file", "command", "error" over
  project-specific coinages.
- When a domain-specific term is unavoidable, define it on first use in one
  clause, then use it consistently.
- Expand an acronym the first time it appears ("continuous integration (CI)").
- Avoid internal code names and team shorthand the reader cannot be expected to
  know.
- Choose one term per concept and stick to it; do not alternate between
  "parameter" and "argument" for the same thing within a document.
- Match the casing and spelling of identifiers exactly when you do name them,
  even while keeping the surrounding prose plain.
- Aim for words a competent developer new to this project would understand on
  first read.
